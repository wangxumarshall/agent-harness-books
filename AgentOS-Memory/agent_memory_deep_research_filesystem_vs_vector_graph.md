# Agent Memory 深度研究：Filesystem-like vs Vector/Graph-like

> **研究目标**：将 Agent Memory 系统分为两大类——Filesystem-like（文件系统范式）和 Vector/Graph-like（向量库/知识图谱/超图范式），深度研究每类的真实案例、技术方案、技术效果、优劣势、应用场景，以及对 CortexMem 创新方案的借鉴。
>
> **数据可信度标注**：★=独立验证 ★☆=论文/官方自报 ☆=项目自报无独立验证 ☆☆=独立复现失败/严重存疑 ⚠️=部分存在/待确认
>
> **研究方法**：基于项目已有 5 轮研究材料 + WebSearch/arXiv/GitHub 一手来源交叉验证

---

## 第一部分：Filesystem-like Agent Memory

> **定义**：以人类可读的文件/目录/Markdown 为核心存储范式，向量索引为辅助检索层的 Agent Memory 系统。核心特征是"源头真相在文件，向量是影子"。

### 1.1 OpenViking（volcengine/OpenViking）—— viking:// 虚拟文件系统范式

**问题背景与真实案例**

火山引擎（字节跳动）内部开发，面向 OpenClaw/VikingBot 等 Agent 框架。GitHub ~4.8K Stars。

- **真实案例**：编码 Agent 跨会话遗忘项目上下文——Agent 记忆分散（记忆在代码、资源在向量库、技能散落），检索黑箱、上下文不可分层、无法自我迭代。OpenViking 通过 `viking://` 协议统一管理后，声称完成率提升 43-49%。
- **用户痛点**：传统 Agent 的记忆、资源、技能分别存储在不同系统中，无法统一检索和管理，导致 token 成本高、调试难、长期记忆弱。

**技术方案**

```
viking://
├── memory/        (session/task/long_term)
├── resources/     (docs/code/knowledge)
└── skills/        (tools/workflows)
```

- **存储范式**：虚拟文件系统（`viking://` 协议）+ 向量预滤
- **三层加载策略**：
  - L0：核心摘要 ~100 tokens（始终加载）
  - L1：结构化摘要 ~2K tokens（按需加载）
  - L2：完整原始内容（精确查询时）
- **检索机制**：意图分析 + 向量预滤 + 目录递归探索 + 可视化轨迹
- **自进化循环**：Session 自动压缩对话、工具调用，异步提取长期记忆并更新 User/Agent 目录
- **技术栈**：Rust CLI + VLM/Embedding 模型

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| Token 节省 | 83-91% | ☆ 项目自报，基线疑为全量注入 |
| 任务完成率提升 | +43-49% | ☆ 项目自报 |
| LoCoMo | ~49%（提升后） | ☆ |

**优劣势**

- **优势**：① `viking://` URI 可寻址——唯一实现 URI 级记忆寻址 ② 统一管理记忆/资源/技能三类上下文 ③ 自进化循环
- **劣势**：① AGPL-3.0 许可证（与官网声称 Apache 2.0 矛盾），商用受限 ② 依赖 VLM/Embedding，部署复杂度中高 ③ 早期项目，工程成熟度低 ④ 性能数据全部自报

**应用场景**

- **最适合**：编码 Agent（需要统一管理代码资源+记忆+技能）、需要 URI 级可观测性的场景
- **不适合**：轻量级个人助手（部署复杂度高）、安全敏感场景（无安全机制）

**对 CortexMem 的借鉴**

1. **URI 寻址**理念——记忆条目应有唯一标识符，支持精确引用和追溯
2. **三层加载策略**（L0/L1/L2）已被多个系统验证，是 token 效率的核心设计
3. **统一管理记忆/资源/技能**的"上下文工程"视角比纯"记忆"视角更完整

### 1.2 memsearch（zilliztech/memsearch）—— Markdown-first + Milvus 影子索引

**问题背景与真实案例**

Zilliz（Milvus 母公司）开发，面向编码 Agent 生态。支持 Claude Code、OpenClaw、OpenCode、Codex CLI 等多平台插件。

- **真实案例**：编码 Agent 跨会话遗忘，记忆不可读、不可编辑、跨平台不共享。开发者通过每日 Markdown 日志手动编辑记忆，同时享受向量检索的便利。
- **用户痛点**：传统 RAG 缺乏渐进式检索和去重，记忆黑箱无法人工干预。

**技术方案**

```
memory/
├── MEMORY.md              # 手写的长期记忆
├── 2026-02-09.md          # 每日工作日志
└── .memsearch/            # 向量索引缓存
```

- **存储范式**：Markdown 为源（Source of Truth），Milvus 为派生索引
- **三层渐进检索**：L1 Search（rank chunks）→ L2 Expand（full section）→ L3 Transcript（parse raw JSONL）
- **混合搜索**：Dense Vector + BM25 + RRF Reranking
- **去重机制**：SHA-256 内容哈希，只对变化内容重新嵌入
- **写入流程**：插件自动捕获 → haiku 摘要追加到当日 Markdown → 文件监听器实时同步 Milvus

**技术效果**

- 无标准基准（LoCoMo/LongMemEval）测试数据
- Token 节省显著（渐进披露），但无量化数据
- 可信度：☆

**优劣势**

- **优势**：① Markdown-first——人类可读可编辑，git 天然版本控制 ② 影子索引——向量索引是 Markdown 的派生物 ③ SHA-256 去重高效 ④ 跨平台零配置 ⑤ MIT 许可
- **劣势**：① 无遗忘机制——记忆只增不减 ② 依赖 Milvus ③ 无结构化推理 ④ 无安全机制

**应用场景**

- **最适合**：编码 Agent（人类可读+git 版本控制+跨平台共享）
- **不适合**：需要结构化推理的运维场景、需要遗忘机制的长期运行场景

**对 CortexMem 的借鉴**

1. **Markdown-first + 向量影子索引**的混合范式是 CortexMem 的核心设计之一——memsearch 验证了可行性
2. **SHA-256 内容去重**是高效的写入优化策略
3. **三层渐进检索**（search → expand → transcript）是 token 效率的关键设计
4. **缺失的遗忘机制和结构化推理**正是 CortexMem 需要补充的

### 1.3 memU（NevaMind-AI/memU）—— 文件系统式分层记忆

**问题背景与真实案例**

NevaMind AI 开发，面向 OpenClaw/Moltbot/Clawdbot 等 24/7 主动 Agent。已有 3 代社区实现（memov.ai → memovyn → memU）。

- **真实案例**：Moltbot 24/7 运行的客服 Agent，通过 memU 的主动式记忆管理自动分类用户偏好和事件。社区扩展包括 memu-cowork（多 Agent 协作）和 memu-sillytavern-extension。
- **用户痛点**：24/7 主动 Agent 需要低成本长期上下文，传统方案 token 爆炸、无法主动捕捉意图。

**技术方案**

```
memory/
├── preferences/    (communication_style.md, topic_interests.md)
├── relationships/  (contacts/, interaction_history/)
├── knowledge/      (domain_expertise/, learned_skills/)
└── context/        (recent_conversations/, pending_tasks/)
```

- **存储范式**：文件目录树 + 符号链接 + PostgreSQL/pgvector
- **五种记忆类型**：[P] Profile / [E] Event / [K] Knowledge / [B] Behavior / [S] Skill
- **检索机制**：RAG + LLM 双模式检索（向量语义检索 + LLM 深度推理）
- **主动式特性**：自动分类、模式检测、上下文预测
- **遗忘机制**：隐式淘汰（低频沉底）——零成本但不精确

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| LoCoMo | 92%+ | ☆☆ 项目自报，无独立验证，无论文支撑 |
| Token 成本 | 降至 1/10 | ☆☆ 基线不明确 |

> 92%+ LoCoMo 远超 GPT-4 基线（<60%），如果属实将是极其显著的提升，但完全缺乏证据支撑。

**优劣势**

- **优势**：① 主动式设计——面向 24/7 主动 Agent ② 五种记忆类型标签 [P/E/K/B/S] ③ RAG + LLM 双模式检索 ④ 社区生态成熟
- **劣势**：① 性能数据完全不可信 ② 遗忘机制最薄弱 ③ 无学术论文 ④ 无安全机制

**应用场景**

- **最适合**：24/7 主动 Agent（需要自动分类和上下文预测）
- **不适合**：需要精确遗忘的场景、需要结构化推理的复杂运维场景

**对 CortexMem 的借鉴**

1. **五种记忆类型标签 [P/E/K/B/S]** 被 CortexMem 的 L1 事实层直接借鉴
2. **主动式设计**（自动分类、模式检测、上下文预测）是 24/7 Agent 的刚需
3. **RAG + LLM 双模式检索**验证了"分级策略"的可行性
4. **遗忘机制的薄弱**是 CortexMem 需要重点改进的方向

### 1.4 Acontext（memodb-io/Acontext）—— Agent Skills as a Memory Layer

**问题背景与真实案例**

MemoDB 开发，GitHub ~3.3K Stars。支持 LangGraph、Claude、AI SDK 等多框架。

- **真实案例**：编码 Agent 完成复杂的调试任务后，Acontext 自动"蒸馏"出技能文件，下次遇到类似问题可直接复用。任务完成/失败后触发"蒸馏"学习，LLM 推断什么有效/失败及用户偏好。
- **用户痛点**：Agent 完成任务后的经验无法复用，每次"从零开始"。

**技术方案**

- **核心理念**：Skill is Memory, Memory is Skill
- **存储范式**：Markdown 技能文件（人类可读）
- **存储流程**：Session 消息 → 任务完成/失败 → 蒸馏 → 技能代理 → 更新技能
- **检索流程**：任意 Agent → list_skills/get_skill → 渐进式披露（**无向量搜索**）
- **关键创新**：非嵌入检索——使用工具调用和推理进行渐进式披露
- **技术栈**：TypeScript + Python SDK，FastAPI + PostgreSQL + S3 + Redis + RabbitMQ

**技术效果**

- 无标准基准测试数据
- 核心价值体现在定性体验（技能复用减少重复工作）

**优劣势**

- **优势**：① "技能即记忆"范式——唯一将记忆定义为可执行技能的系统 ② 非嵌入检索 ③ 人类可读 ④ 框架无关 ⑤ 无供应商锁定
- **劣势**：① 无向量检索——大规模技能库检索效率不足 ② 无遗忘机制 ③ 蒸馏质量依赖 LLM ④ 技能冲突无仲裁 ⑤ 后端依赖重

**应用场景**

- **最适合**：技能密集型 Agent（编码、运维、工作流自动化）
- **不适合**：需要大规模语义检索的场景、需要精确遗忘的场景

**对 CortexMem 的借鉴**

1. **"技能即记忆"范式**是 CortexMem L3 程序性记忆层的设计来源
2. **蒸馏学习**（任务完成后自动提取技能）是离线学习的优秀实践
3. **渐进式披露**（list_skills → get_skill → 详情）是 token 效率的关键设计
4. **缺失的向量检索**提示 CortexMem 应在技能层补充语义检索能力

### 1.5 Voyager（MineDojo/Voyager）—— 可执行代码技能库

**问题背景与真实案例**

NVIDIA/Stanford 联合开发，MIT 许可。论文 arXiv:2305.16291，高引用量。

- **真实案例**：Voyager 在 Minecraft 中通过不断探索和技能积累，解锁科技树里程碑的速度比基线快 15.3 倍。获得 3.3 倍更多独特物品，旅行距离增加 2.3 倍。
- **用户痛点**：开放式探索 Agent 缺乏持久技能，每次"从零开始"。

**技术方案**

- **三大组件**：① 自动课程（最大化探索的自动课程规划）② 技能库（可执行代码技能的持久存储）③ 迭代提示（结合环境反馈、执行错误和自验证）
- **存储范式**：可执行 JavaScript 代码技能 + 向量检索
- **迭代学习循环**：任务 → 课程代理生成子目标 → 行动代理生成代码 → 环境执行 → 评论代理验证 → 失败则反思改进 → 成功则存入技能库
- **技能格式**：JavaScript 函数，解释性强、可组合、时序扩展

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| 独特物品获得 | 3.3x 更多 | ★☆ 论文自报 |
| 科技树里程碑 | 15.3x 更快 | ★☆ 论文自报 |

> 数据在 Minecraft 环境中验证，不可直接外推到其他场景。

**优劣势**

- **优势**：① 可执行代码技能——记忆不是文本描述，而是可直接执行的代码 ② 迭代学习闭环（Plan→Act→Verify→Reflect→Store） ③ 环境反馈驱动——技能质量由执行结果验证 ④ 学术验证充分
- **劣势**：① 场景绑定——迁移到编码/运维场景需完全重写 ② 技能冲突无仲裁 ③ 无遗忘机制 ④ 代码安全性风险 ⑤ 非对话记忆系统

**应用场景**

- **最适合**：开放式探索 Agent（游戏、自动化测试、科学发现）
- **不适合**：对话记忆（用户画像、偏好）、需要精确遗忘的场景

**对 CortexMem 的借鉴**

1. **可执行代码技能**是程序性记忆的最佳形态——CortexMem L3 层应支持代码/脚本/工作流作为记忆
2. **迭代学习闭环**（Plan→Act→Verify→Reflect→Store）是 CortexMem 离线学习的设计蓝本
3. **环境反馈验证**比 LLM 主观判断更可靠——CortexMem 的技能验证应优先使用环境执行结果
4. **场景迁移困难**提示 CortexMem 需要设计**技能抽象层**，使技能可跨场景复用

### 1.6 xiaoclaw-memory（huafenchi/xiaoclaw-memory）—— 零成本纯 Markdown

**问题背景与真实案例**

huafenchi（小花池）开发，MIT 许可。受 memU 启发，面向 24/7 AI Agent。

- **真实案例**：个人开发者在 $5 VPS 上运行 24/7 Agent，使用 xiaoclaw-memory 无需任何外部依赖。Agent 自行判断记忆类型并标注 [P/E/K/B/S] 标签，无需额外 LLM 调用。
- **用户痛点**：memU/mem0 等系统依赖外部 DB/LLM，部署成本高。

**技术方案**

```
L0: MEMORY.md — 索引 + 统计 + 摘要（~200 tokens）
L1: 主题文件（按类型分类）
    [P] profile.md   [E] events.md   [K] knowledge.md
    [B] behaviors.md  [S] skills.md   [!] security.md
L2: YYYY-MM-DD.md — 每日原始日志（只读最近2天）
```

- **存储范式**：纯 Markdown 文件
- **检索机制**：文件系统浏览 + LLM 直接读取（无向量索引）
- **关键创新**：Agent 自行判断记忆类型，无需额外 LLM 调用（**零额外 token 成本**）

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| 额外 Token 成本 | 0（vs memU 的 3-5 次/轮） | ★ 源码可验证 |
| 外部依赖 | 无 | ★ 源码可验证 |

**优劣势**

- **优势**：① 零外部依赖 ② 零额外 token 成本 ③ 极简部署 ④ 人类可读 ⑤ MIT 许可
- **劣势**：① 无语义检索 ② 无遗忘机制 ③ 无结构化推理 ④ 自分类准确性依赖 Agent 能力 ⑤ 安全标签 [!] 仅是标记

**应用场景**

- **最适合**：个人轻量 Agent（$5 VPS、零配置）、原型验证
- **不适合**：大规模记忆、需要语义检索、需要结构化推理的场景

**对 CortexMem 的借鉴**

1. **零成本自分类标签**是 CortexMem "零成本蒸馏"设计的直接来源
2. **三层架构**（L0 索引 + L1 主题 + L2 日志）验证了分层加载的 token 效率
3. **纯 Markdown 的局限性**（无语义检索、无遗忘、无结构化推理）正是 CortexMem 需要突破的天花板
4. **安全标签 [!] 的概念**可扩展为 CortexMem 的信任级别标记

### 1.7 Memoria（matrixorigin/Memoria）—— Git for Memory, CoW 版本控制

**问题背景与真实案例**

矩阵起源（MatrixOrigin）开发，GTC 2026 发布。GitHub ~193 Stars，InfoQ/CSDN 等多家媒体报道确认。

- **真实案例**：企业运维 Agent 的记忆被环境注入投毒（eTAMP 攻击），通过 Memoria 的版本回滚恢复到安全状态。多人协作时记忆冲突可通过分支/合并解决。
- **用户痛点**：Agent 记忆一旦被投毒或损坏，无法回退到安全状态；记忆变更不可追溯、不可回滚；多人协作时记忆冲突无法解决。

**技术方案**

- **核心理念**：Git for Memory——将软件工程的版本控制最佳实践应用于 Agent 记忆
- **存储范式**：Copy-on-Write (CoW) 数据库（基于 MatrixOne）
- **核心机制**：① Copy-on-Write（每次变更创建新版本，不覆盖旧版本）② 快照/分支/合并 ③ 回滚（发现记忆被投毒时，可回退到已知安全版本）
- **明确批评 Markdown**：指出 Markdown 的"静态、单向、无结构、无法回滚"四大局限
- **技术栈**：Rust + MatrixOne 数据库

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| 版本回滚能力 | 已实现 | ★ InfoQ 验证 |
| CoW 性能开销 | 待验证 | - |

**优劣势**

- **优势**：① 版本回滚——唯一支持记忆版本控制和回滚的系统 ② CoW 零拷贝快照 ③ 分支/合并支持 A/B 测试 ④ 与 Git 生态兼容
- **劣势**：① CoW 存储开销——每次变更都创建新版本 ② 分支合并冲突策略待完善 ③ 依赖 MatrixOne ④ 工程成熟度低 ⑤ 无语义检索

**应用场景**

- **最适合**：安全敏感场景（需要投毒防御和回滚）、多人协作记忆管理
- **不适合**：轻量级个人助手（部署复杂度高）、需要大规模语义检索的场景

**对 CortexMem 的借鉴**

1. **版本回滚**是 CortexMem 安全层的关键补充——形成"预防（防火墙）+ 检测（溯源）+ 隔离（沙箱）+ 恢复（版本回滚）"的完整安全闭环
2. **对 Markdown 四大局限的批评**提示 CortexMem 需要在 Markdown-first 基础上增加结构化和版本控制

### 1.8 lossless-claw（Martian-Engineering/lossless-claw）—— DAG 摘要系统

**问题背景与真实案例**

Martian Engineering 开发，MIT 许可。OpenClaw 生态的 LCM（Lossless Context Management）插件。

- **真实案例**：编码 Agent 在长会话中需要回溯到 100 条消息前的某个具体代码讨论，lossless-claw 通过 DAG 摘要展开恢复原文。OOLONG 基准 74.8 分 vs Claude Code 70.3 分。
- **用户痛点**：传统上下文压缩会丢失信息，Agent 无法回溯到原始对话。

**技术方案**

```
SQLite DB
├── messages/           # 原始消息（永不丢失）
├── leaf_summaries/    # 叶子层摘要
├── d1_summaries/      # 一级摘要
└── d2_summaries/      # 二级摘要
```

- **存储范式**：SQLite + DAG 层级摘要
- **上下文组装**：Fresh Tail（最近 N 条原始消息）+ Summary DAG（层级摘要结构）+ 动态展开（`lcm_expand`）
- **检索工具**：`lcm_grep`（全文搜索）、`lcm_describe`（摘要详情）、`lcm_expand`（展开恢复原文）
- **关键原则**：原始消息永不删除

**技术效果**

| 指标 | 数据 | 可信度 |
|------|------|--------|
| OOLONG 基准 | 74.8 分 | ☆ 项目自报 |
| vs Claude Code | 70.3 分 | ☆ 项目自报 |
| 信息无损 | 原始消息永不删除 | ★ 源码可验证 |

**优劣势**

- **优势**：① 无损上下文管理——原始消息永不删除 ② DAG 层级摘要比线性摘要更灵活 ③ 增量压缩 ④ 简单工具集
- **劣势**：① 死记忆——只存不更新 ② 无语义检索 ③ 无结构化推理 ④ 摘要质量依赖 LLM ⑤ 存储膨胀

**应用场景**

- **最适合**：需要完整审计追踪的场景（合规、法律）、需要回溯原始对话的编码 Agent
- **不适合**：需要遗忘机制的场景、需要语义检索的大规模记忆

**对 CortexMem 的借鉴**

1. **"原始消息永不删除"**原则——CortexMem L0 Raw Episodes 层应采用 append-only event log
2. **DAG 层级摘要**是 CortexMem 后台 consolidation 路径的设计参考
3. **动态展开**（lcm_expand）是渐进披露的优秀实践
4. **"死记忆"的局限**提示 CortexMem 必须实现反思/融合/遗忘机制

### 2.1 mem0（mem0ai/mem0）—— 通用记忆层 SDK，LLM 决策机

**问题背景与真实案例**

GitHub ~52K Stars，PyPI 下载量 14M+，ECAI 2025 论文（arXiv:2504.19413）。当前 Agent Memory 领域社区规模最大的项目。

- **真实案例**：① 个性化 AI 助手——记住用户偏好（"我喜欢深色主题"）② 客服 Agent——记住历史交互（"上次购买了产品 A"）③ 编码 Agent——记住项目上下文（"项目使用 React + TypeScript"）。已集成到 Claude Code、Cursor、LangGraph 等主流框架。

**优劣势**

- **优势**：① 社区生态最强（52K stars, 14M+ 下载），工程成熟度最高 ② 一行 SDK 集成 ③ 托管+自托管双模式 ④ v1.0 扩展了 graph memory

**应用场景**

### 2.6 TiMem（arXiv:2601.02845）—— 5 层时序记忆树 + CLS 理论

**问题背景与真实案例**

中科院自动化所，已有商业 API 服务 timem.cloud。

- **真实案例**：长期对话 Agent 需要时间层次化记忆——用户今天说的"我换了工作"需要覆盖上周的"我在 A 公司"，但不应完全删除旧信息。TiMem 的 5 层 TMT 实现了从原始事实到稳定画像的渐进巩固。
- **用户痛点**：现有记忆系统缺乏时间层次结构，无法区分事实、会话、日、周、画像等不同时间尺度的记忆；检索时无法根据问题复杂度自适应调整检索深度。

**技术方案**

- **核心架构**：5 层时序记忆树（Temporal Memory Tree, TMT）+ 复杂度感知召回
- **5 层 TMT**：
  - L1 事实层：原始事实片段
  - L2 会话层：会话摘要
  - L3 日层：日模式归纳
  - L4 周层：周趋势
  - L5 画像层：稳定人格画像
- **复杂度感知召回**：简单问题只检索 L1/L2，复杂问题才往 L3/L4/L5 找，**无需 LLM 决策**即可实现分层路由
- **理论根基**：互补学习系统理论（CLS）——模拟海马体→新皮层转移

**技术效果**

| 指标 | 数值 | 来源 | 可信度 |
|------|------|------|--------|
| LoCoMo | 75.30% | 论文自报 | ★☆ |
| LongMemEval-S | 76.88% | 论文自报 | ★☆ |
| Token 节省 | 52.2% | 论文自报 | ★☆ |

> TiMem 的贡献不仅是数值，更在于把"层次化 consolidation"说清楚了——低层保留细节，高层保留抽象画像，查询复杂度决定走哪一层。

**优劣势**

- **优势**：① CLS 理论提供完整的认知科学框架 ② 复杂度感知召回无需 LLM 决策——零成本分层路由 ③ 5 层 TMT 结构清晰 ④ 部署成本低 ⑤ 已有商业 API
- **劣势**：① 5 层归纳仍需 LLM 调用 ② 时序树的层级划分可能不适用于所有场景 ③ LoCoMo 75.30% 在竞品中处于中游 ④ 缺乏 L0 感知层

**应用场景**

- **最适合**：长期对话 Agent、陪伴型 AI、需要时间层次化记忆的场景
- **不适合**：编码 Agent（记忆不按日/周组织）、实时性要求极高的场景

**对 CortexMem 的借鉴**

1. **5 层 TMT 是 CortexMem 分层架构的直接参考**
2. **复杂度感知召回**：零成本分层路由是 CortexMem 三级检索路径的理论基础
3. **CLS 理论**：比 Ebbinghaus 遗忘曲线更完整的认知科学框架
4. **改进空间**：TiMem 缺乏 L0 感知层和安全治理，这正是 CortexMem 的差异化方向

### 2.7 LiCoMemory（arXiv:2511.01448）—— CogniGraph 轻量认知图谱

**问题背景与真实案例**

HKUST + 华为香港研究中心 + WeBank。GitHub: EverM0re/LiCoMemory。

- **真实案例**：需要在图谱表达力和轻量部署之间取得平衡的长期对话场景。传统知识图谱构建成本高、更新延迟大；扁平向量检索缺乏结构化推理能力。
- **用户痛点**：知识图谱过重、语义与拓扑缠绕太深、更新与检索都变慢。

**技术方案**

- **核心架构**：CogniGraph——轻量层次认知图谱
- **存储范式**：轻量图 + 向量混合
- **检索机制**：时序+层次感知搜索 + 集成重排序
- **核心理念**：图不是用来做全量推理的，而是用来做检索导航的

**技术效果**

- 论文报告在 LoCoMo 和 LongMemEval 上超越 baselines
- 显著降低更新延迟
- 可信度：★☆（论文自报）

**优劣势**

- **优势**：① 轻量图谱是工程上的重要创新 ② "图即索引"理念 ③ 更新延迟显著降低 ④ 多路检索+重排序
- **劣势**：① CogniGraph 表达力弱于完整知识图谱/超图 ② 论文预印本未经 peer review ③ 缺乏安全治理机制

**应用场景**

- **最适合**：需要结构化推理但无法承担重型知识图谱部署成本的场景
- **不适合**：需要超图级 n-元关系表达的场景、纯向量检索即可满足的简单场景

**对 CortexMem 的借鉴**

1. **CogniGraph 的"图即索引"理念**：图不应承担全量推理，而应作为检索导航层
2. **轻量部署**：在表达力和部署成本之间找平衡是工程落地的关键
3. **时序+层次感知搜索**：时间衰减 + 层级权重的检索策略值得借鉴

EverMind AI 团队，GitHub ~3.3K Stars。

- **存储范式**：超图（HyperMem）——支持 n-元关系表达
- **三大组件**：MemCells（基本记忆单元）+ MemScenes（场景级记忆组织）+ EverCore（记忆 OS 调度器）
- **engram 生命周期**：Episodic trace formation → Semantic consolidation → Reconstructive recollection

**技术效果**

| HyperMem LoCoMo | 92.73% | 未独立验证 | ☆（同团队） |

**优劣势**

- **优势**：① 概念最完整——engram 生命周期 + 超图 + 自组织调度 ② 超图 n-元关系表达 ③ EverCore 热度调度
- **劣势**：① **不可复现**——独立复现仅 38.38% ② 部署复杂度高 ③ 同团队循环引用 ④ 超图计算开销大 ⑤ 工程成熟度低

**应用场景**

- **最适合**：学术研究、概念验证
- **不适合**：生产部署（数据不可信）、成本敏感场景

**对 CortexMem 的借鉴**

1. **engram 生命周期理念**：记忆的形成→巩固→回忆三阶段是 CortexMem 巩固引擎的理论基础
2. **超图 n-元关系**：比普通图更强的关系建模——但需注意计算开销
3. **反面教训（最重要）**：不可复现的 SOTA 声称会严重损害项目信誉——CortexMem 必须提供可复现的 benchmark 结果
4. **反面教训**：概念完整不等于工程可行——过度复杂的架构可能导致工程无法落地

### 2.9 MIRIX——6 类记忆 + Active Retrieval

**问题背景与真实案例**

UCSD + NYU 团队，有 GitHub 仓库、Mac 应用、详细论文（arXiv:2507.07957）。

- **真实案例**：多智能体协作场景——不同 Agent 负责不同记忆类型（如一个 Agent 管理语义记忆，另一个管理程序性记忆），通过 Active Retrieval 主动关联所有记忆类型。
- **用户痛点**：单一记忆类型无法覆盖 Agent 的多样化需求；被动检索导致 Agent 重复查询相同信息。

**技术方案**

- **核心架构**：6 类记忆 + Active Retrieval + 多智能体协同
- **6 种记忆类型**：Core Memory、Episodic Memory、Semantic Memory、Procedural Memory、Resource Memory、Knowledge Vault
- **Active Retrieval**：Agent 不被动等待查询，而是**主动关联所有记忆类型**——用户输入自动触发跨类型检索
- **效果**：减少 87% 的 API 调用

**技术效果**

| 指标 | 数值 | 来源 | 可信度 |
|------|------|------|--------|
| LoCoMo | 85.4% | 论文自报 | ★☆ |
| API 调用减少 | 87% | 论文自报 | ★☆ |

**优劣势**

- **优势**：① 6 类记忆覆盖最全面 ② Active Retrieval 减少重复查询 ③ 多智能体协同记忆管理 ④ LoCoMo 85.4% 在已验证系统中处于第一梯队
- **劣势**：① 6 类记忆的分类和维护成本高 ② Active Retrieval 可能引入噪声 ③ 多智能体协调增加系统复杂度 ④ 缺乏安全治理

**应用场景**

- **最适合**：多智能体协作场景、需要多种记忆类型协同的复杂 Agent
- **不适合**：简单场景（6 类记忆过度设计）、单 Agent 场景

**对 CortexMem 的借鉴**

1. **6 类记忆分类**：CortexMem 的 P/E/K/B/S 五类自分类直接参考了 MIRIX 的多类型设计
2. **Active Retrieval**：主动关联检索是 CortexMem Active Retrieval 的直接来源——但需配合过滤机制避免噪声
3. **87% API 调用减少**：Active Retrieval 的工程价值已被验证

### 2.10 MemBrain（Feeling AI）—— LLM 亲和记忆 + 子 Agent 协调

**问题背景与真实案例**

Feeling AI，创始人戴勃。GitHub ~269 stars，v1.5。新智元、机器之心等多家媒体报道确认。

- **真实案例**：需要"活人感"记忆的陪伴型 AI。MemBrain 的"中庸遗忘"——重要的事保留，细节模糊，旧经历融入新反应——比规则式遗忘更接近人类记忆。
- **用户痛点**：传统记忆系统采用线性流程（提取→存储→检索→注入），图算法与大语言模型之间存在范式差异——图算法在算的时候，LLM 只能站在旁边看着。

**技术方案**

- **核心架构**：LLM 亲和记忆（LLM-Affinitive Memory）+ 子 Agent 自主协调
- **关键创新**：
  - 打破线性流程：记忆提取、存储、检索由子 Agent 自主协调，非固定流水线
  - LLM 直接参与记忆推理：让模型在记忆空间中直接推理，而非图算法算完再喂给 LLM
  - 严控时间戳：所有记忆附加精确时间信息
  - "中庸"遗忘：模拟人类的"活人感"

**技术效果**

| 指标 | 数值 | 来源 | 可信度 |
|------|------|------|--------|
| LoCoMo | 93.25% | 自报 | ☆（无正式论文） |
| LongMemEval | 84.6% | 自报 | ☆ |
| PersonaMem-v2 | 51.50% | 自报 | ☆ |
| KnowMeBench Level III | +300% | 自报 | ☆ |

> **关键注意**：所有数据均为自报，无正式论文，无独立验证。LoCoMo 93.25% 如果属实则是 SOTA，但可信度极低。

**优劣势**

- **优势**：① LLM 亲和记忆是范式创新 ② "活人感"遗忘比规则式更自然 ③ 子 Agent 协调打破线性流程 ④ 如果数据属实，LoCoMo 93.25% 是 SOTA
- **劣势**：① 数据可信度极低——无正式论文 ② 子 Agent 协调增加系统复杂度 ③ LLM 直接参与记忆推理增加 token 成本 ④ 工程成熟度低

**应用场景**

- **最适合**：陪伴型 AI、需要"活人感"记忆的场景
- **不适合**：事实密集型场景、成本敏感场景

**对 CortexMem 的借鉴**

1. **LLM 亲和记忆理念**：CortexMem 的三级检索路径中，"推理"级别直接借鉴了 MemBrain 的 LLM 空间推理理念
2. **"活人感"遗忘**：CortexMem 的遗忘曲线 + CLS 引擎应融入"中庸遗忘"理念——不是简单删除，而是模糊细节、保留核心
3. **子 Agent 协调**：CortexMem 的多 Agent 总线可借鉴子 Agent 自主协调模式
4. **反面教训**：无正式论文的自报 SOTA 不可信——CortexMem 必须提供可复现的 benchmark

## 第三部分：两大范式深度对比与 CortexMem 借鉴总结

### 3.1 Filesystem-like vs Vector/Graph-like 核心差异

| 维度 | Filesystem-like | Vector/Graph-like |
|------|----------------|-------------------|
| **核心存储** | 人类可读文件/Markdown | 向量/图/超图索引 |
| **检索方式** | 文件浏览 + 向量辅助 | 向量语义 + 图遍历 + 超图推理 |
| **可观测性** | 极强（人类可读可编辑） | 弱到中（需 API/面板查看） |
| **结构化推理** | 弱（目录层次/标签） | 强（图遍历/超图/LLM空间推理） |
| **部署复杂度** | 低到极低 | 中到高 |
| **遗忘机制** | 几乎无 | 部分有（TiMem CLS、EverMemOS engram） |
| **安全治理** | 仅 Memoria 有版本回滚 | 几乎无 |
| **Token 效率** | 高（分层加载/渐进披露） | 中到高（稀疏检索） |
| **适用场景** | 编码Agent、个人轻量Agent | 长期对话Agent、多Agent协作、复杂推理 |

### 3.2 两大范式的共同局限

1. **遗忘机制普遍缺失**：18 个系统中，仅 TiMem（CLS 巩固）、EverMemOS（engram 生命周期，但不可复现）、Memoria（版本回滚）有某种形式的遗忘/治理
2. **安全治理几乎为零**：仅 Memoria 有版本回滚，eion 有命名空间隔离，其余均无
3. **评估不可信**：几乎所有性能数据都是自报，缺乏统一基准下的公平对比
4. **记忆投毒无防**：AgentPoison 攻击成功率 85%+，eTAMP 跨站环境注入 ASR 32.5%，但无系统有防御机制

### 3.4 CortexMem 的三个真正独特维度（经全网验证确认）

经过对 18 个系统的深度对比，CortexMem 的**不可复制差异化**是：

1. **安全内生四层闭环**（防火墙+溯源+沙箱+版本回滚）——**无竞品对标**
   - Memoria 仅有版本回滚（恢复层），无预防/检测/隔离
   - 其余系统均无安全治理

2. **全认知栈覆盖**（L0 感知 KV Cache → L4 画像）——**唯一覆盖感知层**
   - TiMem 覆盖 L1-L5 但无 L0 感知层
   - 其余系统均从 L1 开始

3. **Markdown-first + 版本控制 + 分层调度三位一体**——**竞品仅各具其一**
   - memsearch 有 Markdown 无版本
   - Memoria 有版本无 Markdown
   - OpenViking 有分层无版本

这三个维度是 CortexMem 相对所有竞品的**不可复制差异化**，也是技术方案设计的核心锚点。
